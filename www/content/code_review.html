Code Review Procedure

The code review procedure describes how new or changed code and documentation are reviewed and accepted for commitment to the project repository. The code review is the central part of Phase 6, Implementation and Documentation Review Phase, of the development process.

Each contribution must pass a code review before being committed to the repository. The goal of code reviews is to ensure that only high-quality code with complete tests and documentation enters the repository.

Because committers are responsible for all their commitments to the repository, they must choose the level of detail for a particular code review. This means that small or trivial code changes can go through a simplified review, where complex code will likely require a thorough review that includes most, if not all, steps.

The code review procedure is a best practice guideline that evolves over time.

Roles

Reviewers
Reviewers are typically the committer plus additional community members with the appropriate expertise. The committer acts as the review moderator and is responsible for driving the review to completion.
 
Author
The author is typically the contributor of the code. The author provides the materials for review, answers questions, and implements changes that result from the review.
 

Code Review Procedure

The author performs the following tasks:

  1. Create and submit the contribution as described in Submitting Contributions.

  2. If the contribution addresses a CR, set the CR state to Review in Issue Tracker.

  3. Work with the reviewers during the review, if necessary, to answer questions or provide information.

  4. After the review, check the review results. Depending on the results, the contribution might be accepted, minor modifications might be indicated, major changes might be necessary, or the contribution might be rejected.

  5. If modifications are required, fix the indicated problems. Add comments to the source or update the documentation for any issues that the review might have uncovered. Also, update the change description appropriately.

  6. After all review comments are addressed, make sure a current change package is generated and is ready to be committed.

Reviewers perform the following tasks:

  1. Expand the zip or tar file that contains the contribution.

  2. Read the change description.

  3. View the code using the Subversion diff command or other method of your choice.

  4. Work through the questions in the code review check list.

  5. Fill in the code review output template and forward it to the author.

  6. If the code review result is Accepted As Is or Accepted With Minor Modifications, close the code review and enter the end date in the code review output template.

Code Review Checklist

  • Is there information in the required fields?

  • Do you understand the code?

  • Is the code appropriate for the current milestone or project status?

  • Do all of the changes implement a required feature or fix the bug as listed in the change description?

  • Are there test cases for the code changes or new code?

  • Have all applicable tests been executed with the expected results?

  • Are the coding standards of the project met?

  • Does the code and documentation match the design, specification, or requirements documents, if any?

  • Is the code well documented and maintainable?

  • Is the documentation complete and correct?

  • List defects or missing test cases.

  • Fill out the code review output template.

Code Review Output Template

Provide the following information:

  • Reviewer name

  • Review details

  • Reviewed item

  • Version, if applicable

  • End date of review

  • Findings

  • Location of reviewed code

  • Comments

  • Severity (severity levels are the same as for CRs)

  • Review result

    • Accepted as is
    • Accepted with minor modifications
    • Major modifications and another review required
    • Rejected (total rework needed)

Committing Changes

Before you commit a change package, the following prerequisites must be met:

  • The code review must be complete with a result of Accepted or Accepted With Minor Modifications and the appropriate changes must be made.
  • A current changes page must be generated.
  • The pre-commit checklist must be verified.

The committer can now commit the change package to the repository using the project-specific configuration management procedures and tools.

If the contribution addresses a CR, set the CR state to Committed in Issue Tracker.
