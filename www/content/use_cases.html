Contribution Use Cases

This section describes the high-level flow of two sample contributions through the development and contribution process. Concepts of the development process are explained in the Development Process. A detailed understanding of these concepts is not necessary for understanding the use cases, which are intended to give you a feel for how the process work.

Use Case A: Simple Contribution

This example demonstrates the simplest possibly case, which is a one-line change to a comment in a single source file. The change is obviously trivial and requires no testing other than to ensure that the required configurations build.

The following steps describe the contribution for this use case:

  1. The contributor finds a committer involved in the appropriate module with whom to work.

  2. Because the change is localized and trivial, the committer and the contributor agree that none of the first three phases of the development process apply and therefore can be skipped.

  3. In Phase 4, Implement and Test, the contributor implements the code change. No unit tests or regression tests are necessary. No documentation is affected.

  4. In Phase 5, Integration and System Test, the contributor successfully builds the required configurations. No integration or system tests are necessary.

  5. In Phase 6, Implementation and Documentation Review, the contributor creates a change package, completes the pre-commit checklist, and submits the code change as a contribution through Issue Tracker.

  6. The committer receives the contribution. Because the code is trivial and the committer has signed the Sun Contributor Agreement, there are no actions to be taken in the committer review flow.

  7. The committer performs a simplified code review, completing the code review output template and closing the review as Accepted As Is.

  8. The committer commits the change package.

Use Case B: Contribution with New Functionality

This example demonstrates a more complex, but typical, contribution that exercises most of the steps in the process. In this use case, the contribution is an idea to create new functionality that combines and builds on existing features of two project modules.

The following steps describe the contribution for this use case:

  1. In Phase 1, Idea and Concept, the contributor discusses the idea on the appropriate forum or mailing list and creates a new RFE.

  2. In Phase 2, Planning and Review, the contributor starts planning the functionality and finds a committer with whom to work. The contributor creates a planning document that is reviewed by the committer and other interested members. The contributor and committer agree that the other steps in the phase can be skipped.

  3. In Phase 3, Design and Review, the contributor creates a design document that is reviewed by the committer and other interested members. The contributor and committer agree that additional cross-functional testing is needed for the new functionality, so the contributor creates a test plan that details three new test cases. None of the other steps of the phase is required.

  4. In Phase 4, Implement and Test, the contributor develops the code and runs unit tests that pass. The new tests that are described in the test plan are also developed and executed by the contributor. The committer advises the contributor that there are required regression and integration tests, so the contributor also runs those tests. Finally, the contributor writes the necessary documentation.

  5. In Phase 5, Integration and System Test, the contributor ensures that his or her repository is up to date and successfully builds all required configurations. The committer advises the contributor on the required integration and system tests, which the contributor also runs.

  6. In Phase 6, Implementation and Documentation Review, the contributor creates a change package, completes the pre-commit checklist, and submits the code as a contribution through Issue Tracker.

  7. The committer receives the contribution and sets the CR to Review. The committer has signed the Sun Contributor Agreement, but the committer review flow indicates that stakeholders are to be consulted to ensure that everyone who is affected is okay with the new code. The committer discussed the change during planning in Phase 2, so there are no objections from any of the stakeholders.

  8. The committer performs a full code review and completes the code review output template. Due to the early involvement of the committer and QA experts, only minor issues are found and the review is closed as Accepted With Minor Modifications.

  9. The contributor updates the code or documentation according to the review results and creates a new change package.

  10. The committer commits the change package and updates the CR to Committed.
